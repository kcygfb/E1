# 回合对战设计文档

更新日期：2026-07-22  
当前阶段：V2 JSON 牌库（8 张牌/4 分类）已接通，流血 tick/处决/统一状态 tick 系统已实现

## 1. 已确认的核心原则

- JSON 是卡牌静态数据的唯一来源，不为每张牌创建一个 `Card Definition.asset`。
- 赛前选牌界面只保存卡牌 ID；进入战斗后，程序按 ID 自动创建 `CardInstance`。
- 卡牌强化发生在战斗中，消耗魔法点；不是赛前选牌时强化。
- 强化状态属于当前战斗里的某一个卡牌实例，不修改 JSON，也不影响同名的另一张牌。
- 玩家没有护盾系统。
- 处决不会提供额外回合；处决是击破韧性后追加的一次伤害结算。
- 处决进入强制确认阶段：玩家必须点击确认，不能跳过确认直接继续出牌或结束回合。

## 2. 数据与实例

### 2.1 静态牌库

牌库位于 `Assets/StreamingAssets/CardDataV2`，由 `manifest.json` 管理 4 个分类文件，共 8 张牌（通过 `copies` 字段实现 15 张牌组）。

静态数据包括：

- 卡牌 ID、中英文名、分类；
- 消耗资源与数量；
- 是否特殊牌；
- 有序效果列表；
- 每个效果的基础数值和强化数值。

JSON 只描述规则数据，不保存“在手牌中、是否已强化、属于哪一场战斗”等运行时状态。

### 2.2 赛前选择

选牌界面从 `CardJsonRepository.Cards` 生成卡牌展示。玩家确定牌组后，只调用：

```csharp
BattleSession.SetSelectedDeck(selectedCardIds);
```

列表允许重复 ID，代表牌组中有多张同名牌。赛前选择数据中不允许携带强化状态。

### 2.3 战斗实例

进入战斗后，每个 ID 会生成独立实例，例如：

- `blade_dagger#01`
- `blade_dagger#02`

两者共享同一份不可变 `CardSpec`，但各自拥有独立的 `IsUpgraded`。

一张牌在本场战斗中强化后，即使进入弃牌堆并重新洗回牌库，仍保持强化；战斗结束后该状态销毁。

## 3. 回合与牌堆流程

初始牌组必须为 15 张，该数值暴露在 `Combat Rules` 中。赛前选牌未满 15 张时 `Begin` 不可用；
来自 `BattleSession` 的牌组数量不一致时，战斗初始化直接失败。只有直接打开战斗场景时使用的
Inspector 调试牌组允许数量不一致并输出警告。

玩家回合开始按以下顺序结算：

1. 进入玩家回合开始阶段；
2. 处理所有敌人的持续状态 tick（流血等）并检查死亡；
3. 重置本回合魔法点消耗额度和魔法牌使用次数；
4. 恢复行动点；
5. 固定抽 4 张牌；
6. 进入玩家输入阶段。

玩家看到的恢复与抽牌可以用并行动画表现，但规则代码必须按固定顺序产生事件，避免异步动画改变结算结果。

玩家结束回合时：

1. 所有未使用手牌进入弃牌堆；
2. 进入敌方回合；
3. 敌方完成行动；
4. 开始下一玩家回合。

抽牌堆不足 4 张时，先抽完剩余牌，再把弃牌堆洗入抽牌堆，继续补足本次抽牌。没有可洗的牌时才停止抽牌。

## 4. 行动点

- 基础行动点：3。
- 每个玩家回合恢复到有效上限。
- `Base Action Points` 暴露在规则资源中。
- 敌人可以通过 `ActionPointModifier` 降低后续回合的恢复量。
- 魔法牌不消耗行动点。

## 5. 魔法点与战斗内强化

根据当前设计图，第一版规则如下：

- 初始魔法点：3。
- 魔法点上限：3。
- 每回合最多消耗 1 点魔法点。
- “强化基础牌”和“使用魔法牌”共享这 1 点消耗额度。
- 强化一张基础牌消耗 1 点魔法点。
- 魔法牌每回合最多使用 1 张，消耗 JSON 中声明的魔法点，不消耗行动点。
- 每累计消耗 3 点魔法点，自动释放大招。
- 大招造成可配置的大量伤害，并使目标眩晕 1 回合。
- 大招结算后返还 3 点魔法点，返还后不超过上限。
- 大招消耗进度跨回合累计，触发后扣除 3 点进度。

当前把图中的“一回合耐用 1 点”解释为“每回合最多消耗 1 点魔法点”。如果原意不同，需要修改 `Maximum Mana Spend Per Turn` 的语义。

大招伤害数值仍在商讨，因此 `Ultimate Damage` 默认是 0 并暴露在 Inspector。默认状态下大招仍会产生触发事件、眩晕 1 回合、返还 3 点魔法点。

### 5.1 强化操作

强化必须同时满足：

- 当前处于玩家输入阶段；
- 牌在手牌中；
- 牌使用行动点而不是魔法点；
- JSON 至少有一个不同的 `upgraded` 数值；
- 当前实例尚未强化；
- 玩家有足够魔法点；
- 本回合魔法点消耗额度未用完。

强化成功后不出牌、不消耗行动点、不弃牌。之后使用该牌时，每个效果分别读取其强化数值；JSON 中 `upgraded: null` 的字段继续使用基础值。

### 5.2 大招目标

调用强化接口时，UI 可以传入当前锁定敌人 ID；使用魔法牌时使用该牌的目标。目标无效或已死亡时，第一版临时选择第一个存活敌人。多敌人正式顺序仍待确定。

## 6. 卡牌效果结算

效果严格按 JSON 数组顺序执行。

已经进入规则结算的效果：

- 伤害与多段伤害；
- 韧性伤害，支持点数和最大韧性百分比；
- 眩晕；
- 抵消攻击次数；
- 百分比减伤；
- 跳过敌方回合；
- 抽牌；
- 按目标最大生命吸血；
- 获得行动点或魔法点。

已经完成专属规则结算的效果：

- 流血（叠层 + 每回合递减伤害 tick + 暴风雪比例伤害）；

已经可以解析、但专属规则仍待实现的效果：

- 中毒（已预留 tick 接口）；
- 易伤；
- 状态免疫；
- 召唤伙伴；
- 从弃牌堆由玩家选择并打出卡牌。

未实现效果不会被静默忽略，而是产生 `EffectNotImplemented` 事件，方便 UI、日志和测试发现。

当前没有护甲或护盾，所以普通伤害与真实伤害都直接扣生命；两者的类型仍保留在数据中，等防御规则确定后再区分。

临时取整规则：

- 百分比韧性伤害向上取整；
- 百分比减伤后的最终伤害向上取整；
- 100% 减伤结果为 0。

## 7. 韧性与处决

卡牌使敌人韧性从大于 0 变成 0 时：

1. 暂停后续玩家输入；
2. 进入强制处决确认阶段；
3. 玩家确认；
4. 结算一次额外处决伤害；
5. 目标未死亡时恢复韧性；
6. 返回当前玩家回合，不增加回合数、不恢复行动点、不重新抽牌。

当前规则：

- 小兵：处决直接清空剩余生命；
- 精英：额外伤害、眩晕时长待定，均暴露为配置；
- Boss：额外伤害待定，暴露为配置；
- 韧性恢复量与恢复方式待定，暴露为配置。

## 8. 暂定项与待确认项

以下问题尚在商讨，当前只采用临时默认值：

- 精英和 Boss 的处决伤害、眩晕与韧性恢复数值；
- 普通攻击是否需要敌人意图预告；
- 玩家与最后一个敌人同时死亡时的判定；当前暂定玩家失败优先；
- 多敌人的正式行动顺序；
- 多敌人场景下自动大招的正式选目标规则；
- 大招伤害；
- 中毒、易伤、免疫、召唤和弃牌选择的完整规则；
- JSON 目前没有显式 `target` 字段，V2 按效果类型推断“自己/单个敌人”；

## 9. 表现层约束

规则层只产生有序 `CombatEvent`。UI 和动画监听事件，不直接修改生命、韧性、行动点、魔法点、牌堆或强化状态。

这样可以让“行动点恢复、抽牌、状态刷新”在玩家眼里同时播放，又保证代码内部顺序稳定、可测试。
